# Геном и виртуальная машина роста

## Идея: любой набор байт — это код

Геном — просто массив байт (`Uint8Array`). При исполнении каждый байт
превращается в инструкцию по модулю: `опкод = байт % OPS.length`. Аргументы
(для `PUSH`/`SENSE`/`DIR`) берутся из следующего байта тем же приёмом.

Поэтому **любая случайная последовательность байт — корректная программа**, а
мутации никогда не ломают структуру (нет указателей/ссылок, которые надо чинить
после правки). Это и было целью перехода от старого JSON-генома.

```ts
type Genome = {
  code: Uint8Array
}
```

## Опкоды (`OPS`)

Стек-машина. За один тик исполняется до `VM_STEP_BUDGET` шагов; первое успешное
**структурное** действие (`GROW`/`BRANCH`/`SEED`/`SPIKE`/`SHOOT`) завершает шаг.

| Опкод   | Арг | Действие |
|---------|-----|----------|
| `NOP`   | —   | ничего |
| `PUSH`  | 1   | положить литерал 0..1 (`байт % 101 / 100`) на стек |
| `SENSE` | 1   | положить значение сенсора на стек |
| `LT`    | —   | `b < a` → 1/0 |
| `GT`    | —   | `b > a` → 1/0 |
| `AND`   | —   | логическое И (порог 0.5) |
| `OR`    | —   | логическое ИЛИ (порог 0.5) |
| `IF`    | —   | если вершина стека < 0.5 — пропустить следующую инструкцию |
| `DIR`   | 1   | задать направление: `UP`/`DOWN`/`LEFT`/`RIGHT` |
| `GROW`  | —   | вытянуть филамент в `dir`; родитель созревает (в почве → `ROOT`, в воздухе → `STEM`) |
| `BRANCH`| —   | ответвление в `dir`; родитель остаётся меристемой |
| `SEED`  | 1   | сбросить семя в `dir`; аргумент — доля 0..1 от `genomeSeedReserve` |
| `SPIKE` | —   | шип в соседней клетке; родитель остаётся меристемой |
| `SHOOT` | —   | шип на 2 клетки в `dir`; завершает прогон |

## Сенсоры (`SENSORS`)

`ENERGY`, `LIGHT`, `WATER`, `MINERALS`, `DEPTH`, `HEIGHT`, `AGE`, `RANDOM`, `FOREIGN`,
`SHADE`, `SHADE_DIR`, `MINERAL_DIR`, `CROWD_ABOVE`

- Значения нормализованы в 0..1.
- `ENERGY` — доля заполнения энергии всего растения.
- `SHADE` / `SHADE_DIR` — эффективный уровень затенения в клетке / в соседней по
  `DIR` (0 — светло, 1 — глубокая тень; учитывается рассеивание света).
- `MINERAL_DIR` — минералы почвы в клетке по `DIR` (0, если там воздух).
- `CROWD_ABOVE` — доля чужих растений на одну клетку выше в полосе ±5 по X от
  текущей клетки (максимум 11 колонок → 1.0 = вся полоса занята чужими).
- `FOREIGN` — 1, если в клетке по `DIR` чужое растение или семя в почве.
- `DEPTH`/`HEIGHT` — глубина в почве / высота над землёй.
- Одна и та же программа, читая `DEPTH`/`HEIGHT`, растит и корни (вниз/вбок в
  почве), и побег (вверх к свету).

## Исполнение VM

Активны только клетки `SPROUT` (меристемы). Каждый тик для каждой меристемы:

1. `ip = 0`, стек пуст, `dir = cell.dir`.
2. Декодируем `code[ip]`, исполняем, двигаем `ip` (учитывая аргументы).
3. `GROW`/`SEED`/`SHOOT` при успехе завершают прогон. `GROW` к тому же превращает
   родителя в ткань (`ROOT`/`STEM`). `SPIKE` как `BRANCH` может повторяться за тик.
4. `BRANCH` оставляет родителя меристемой, поэтому при остатке бюджета и энергии
   меристема может ответвиться **несколько раз за тик** (если в геноме несколько
   `BRANCH`) — так растение даёт сразу несколько отростков.
5. Если за бюджет шагов ни одно структурное действие не прошло — меристема
   созревает в `ROOT`/`STEM` (чтобы не тратить энергию впустую).

### Бюджет роста за тик

Число структурных действий растения за тик зависит от **избыточной** энергии
(после резерва на содержание):

```
upkeep = Σ MAINTAIN[type] × senescence(age / maxAge)
surplus = max(0, totalEnergy - upkeep × GROW_ENERGY_RESERVE_TICKS - GROW_MIN_ENERGY_RESERVE)
budget = min(MAX_GROW_ACTIONS_CAP,
             MAX_GROW_ACTIONS_PER_PLANT_PER_TICK
               + floor(surplus / ENERGY_PER_EXTRA_GROW_ACTION))
```

Жёсткого лимита клеток нет: большое растение без дохода кроны голодает и
не получает бюджет роста.

Сначала обрабатываются меристемы в почве (до `ROOT_GROW_BUDGET_FRAC` бюджета),
затем надземные — чтобы корни не «голодали» из‑за приоритета кроны.

## Производные параметры

Выводятся прямо из байт кода, без отдельных полей:

- `genomeMaxAge` — из суммы байт, в диапазоне `MIN_AGE..DEFAULT_MAX_AGE`.
- `genomeSeedReserve` — взвешенная сумма байт, 8..17.
- `genomeHue`/`genomeColor` — FNV-хэш кода → стабильный оттенок (похожие геномы
  дают похожий цвет; используется для оценки числа видов).

## Мутации

`mutate(genome, rng)` работает прямо над массивом байт и всегда даёт валидный
геном. Вероятности масштабируются `MUTATION_RATE`:

- точечная замена байта (чаще малый сдвиг ±12, реже полный рандом),
- вставка случайного байта,
- удаление байта,
- дупликация участка (источник новых «органов»),
- инверсия участка.

Длина ограничена `GENOME.MIN_BYTES..MAX_BYTES`.

## Жизнеспособный шаблон

`randomGenome(rng)` с вероятностью 0.85 выдаёт **живучий** `viableTemplateGenome`
(экспорт `hardyTemplateGenome`), иначе — полностью случайные байты. Приоритет
правил (сверху вниз, **первое успешное действие за прогон меристемы**):

1. якорь у поверхности (`BRANCH DOWN`);
2. **побег вверх** (`GROW UP`) — сразу после якоря, до стержневого корня;
3. стержневой корень вниз и боковые корни в почве (`GROW` / `BRANCH`);
4. умеренная боковая крона — только при `ENERGY > ~0.14`;
5. второй побег вверх — только при `ENERGY > ~0.28`;
6. глубокие боковые корни;
7. семя — **в конце**, только при `ENERGY > ~0.42` и достаточной высоте (payload ~0.88).

Нет ранних «лёгких» семян, шипов и отстрелов — они сжигали энергию до роста
корней и листьев. Стартовая энергия проростка — 12 (было 8).

Рост вверх дешевле, чем вниз (`growCost`: UP ×0.7, DOWN ×1.0, вбок ×0.85).

## Сериализация

Геном кодируется компактной hex-строкой (`serializeGenome` /
`deserializeGenome`). `disassemble(genome)` строит читаемый листинг для UI.

## API

- `randomGenome(rng)` — случайный или шаблонный жизнеспособный геном
- `mutate(genome, rng)`
- `serializeGenome` / `deserializeGenome` / `cloneGenome`
- `disassemble(genome)` — листинг инструкций
- `genomeMaxAge` / `genomeSeedReserve` / `genomeHue` / `genomeColor`
- `estimateSpecies(plants)` — оценка числа видов по оттенкам
